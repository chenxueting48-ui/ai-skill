# 网页活动 AI 工作流

这个仓库用于沉淀网页活动产品经理在网页活动 / H5 活动从 0-1 落地过程中的 AI 工作流，包括业务背景、SOP 摘要、同步文档字段结构、提示词、Codex Skill、输出模板和检查清单。

这里不是泛泛介绍 AI 工具的资料库，而是面向产品经理日常工作的使用仓库。适用场景包括：需求沟通、会议记录整理、需求澄清、同步文档补全、同步文档完整性检查、原型与页面逻辑生成、前后端埋点设计、提测前自查、LQA 多语言检查、压测信息收集、域名与对外发布检查、活动复盘。

## 核心原则

- SOP 是流程规则，用于说明网页活动从需求规划到复盘的推进节点和注意事项。
- 同步文档模板是最终产物结构，用于承接活动基础信息、投放链接、原型&设计稿、多语言、任务、抽奖概率、埋点、后端告警、宣发、验收反馈、需求变更和周进度。
- 同步文档是最终信息沉淀载体。后续提示词和 Skill 生成的内容，应尽量对齐同步文档 sheet、字段和填写规则。
- 信息不足时不要脑补，必须明确标记缺失项。
- 检查类输出要区分“已确认 / 待确认 / 风险项 / 建议动作”。

## 推荐使用方式

1. 先看 `业务背景_context/网页活动SOP摘要.md`，确认当前活动处于哪个流程阶段。
2. 再看 `业务背景_context/同步文档字段结构.md`，确认本次输出应该落到哪些同步文档 sheet 和字段。
3. 日常轻量任务使用 `提示词_prompts`，复制到 ChatGPT 或其他 AI 工具中使用。
4. 高频、稳定、规则明确的任务使用 `技能_skills`。
5. 最终要交付或沉淀的内容，优先套用 `输出模板_templates`。

## 核心 Skill

`sync-doc-fill_同步文档补全`：核心 Skill。用于根据活动需求、会议记录、群聊结论、原型说明、页面文案、奖励规则、任务规则、埋点需求或宣发信息，生成可填入活动同步文档的内容。

`sync-doc-check_同步文档检查`：用于检查同步文档完整性，按 13 个 sheet 输出已确认、待确认、风险项和建议动作。

`tracking-design_前后端埋点设计`：用于生成前后端埋点。前端埋点无论如何都需要生成；后端埋点只有用户明确说明“后端定制开发需求”时才生成。

`launch-risk-review_上线风险检查`：用于 inner 回归后、正式对外前检查域名、path、宣发、运维、人力、看板、流量观察和下线风险。

## 提示词编号规则

- `01-02`：需求沟通和信息整理。
- `03-04`：同步文档补全和完整性检查。
- `05-06`：原型页面逻辑和埋点设计。
- `07-10`：提测、LQA、压测、域名和对外发布。
- `11`：活动复盘和数据沉淀。
- 后续新增提示词应尽量顺延编号，不复用已有编号。

## 术语规则

- 仓库内部占位内容使用“待补充”。
- 面向具体活动输出、同步文档检查、缺失信息确认时使用“待确认”。
- 对外产物优先使用“待确认”，避免让需求方误解为 AI 可以自行补齐。
- 禁止脑补活动时间、时区、奖励、概率、库存、道具 ID、奖励号、QPS、后端类型。

## 维护规则

- 不要随意改动目录结构。
- 不要把人的流程约束误写成 AI 可以执行的动作。
- 不要把后台配置需求误写成后端定制开发需求。
- 不要因为活动涉及奖励、任务、抽奖、兑换，就默认生成后端埋点。
- 新增或修改提示词、Skill、模板时，优先对齐同步文档 sheet 和字段。
