# 需求信息提取与澄清

## 使用场景

当需求方提供初步活动想法、玩法描述或背景信息时，用于帮助产品经理提取活动目标、核心玩法、用户路径、页面模块、后台配置依赖、开发依赖、缺失信息和待确认问题。

## 输入信息

- 需求方原始描述
- 活动背景
- 目标用户或地区
- 预期上线时间
- 已知玩法、奖励、任务或页面信息

## 输出要求

- 提取已有信息，不改变需求含义。
- 明确列出缺失信息和待确认问题。
- 区分需求侧待补充和 Web 侧待评估。
- 输出下一步动作。

## 输出格式

```text
# 需求信息提取与澄清

## 已提取信息
## 活动目标
## 核心玩法
## 用户路径
## 页面模块
## 后台配置依赖
## 开发依赖
## 需求侧待补充
## Web 侧待评估
## 下一步动作
```

## 检查规则

- 是否基于输入信息提取，没有自行扩写玩法。
- 是否把缺失信息标记为“待补充”。
- 是否明确哪些问题需要需求方确认。
- 是否列出 Web 侧需要评估的能力点。

## 禁止事项

- 不要替需求方确定奖励、概率、任务规则。
- 不要擅自新增页面或玩法。
- 不要把推测写成已确认。
- 不要输出无法执行的泛泛建议。

## 示例输入

需求方想做一个新用户抽奖活动，用户完成注册后可抽奖，奖励和概率还没定，希望 8 月初上线。

## 示例输出

```text
## 已提取信息
- 活动对象：新用户。
- 核心动作：完成注册后获得抽奖机会。
- 计划上线：8 月初。

## 需求侧待补充
- 奖励内容、概率、库存。
- 新用户定义。
- 活动地区和语言范围。

## Web 侧待评估
- 后台是否支持注册状态识别和抽奖次数配置。
- 是否需要接入奖励发放接口。
```
