# sync-doc-check_同步文档检查

## name

sync-doc-check_同步文档检查

## description

用于检查网页活动 / H5 活动同步文档是否完整，并确认各 sheet 是否具备支撑配置、研发、测试、LQA、压测、上线和复盘的必要信息。同步文档是最终信息沉淀载体，检查结果必须对齐同步文档 sheet 和字段。

## when_to_use

- 同步文档初稿完成后，需要检查缺失项时。
- 提测前需要确认文档是否支持配置、开发和测试时。
- 对外发布前需要回看关键发布信息是否完整时。
- 需求多次变更后，需要梳理风险项和待确认项时。
- 需要确认同步文档是否已覆盖活动同步文档模板的核心 sheet 时。

## rules

- 输出时必须区分“已确认 / 待确认 / 风险项 / 建议动作”。
- 检查结果必须按同步文档 sheet 分组。
- 检查活动基础信息、投放链接、原型&设计稿、页面多语言、活动任务、抽奖概率、用户中奖信息收集、埋点、后端告警、宣发计划、验收反馈、需求变更、每周进度同步。
- 活动时间必须明确到时分秒和时区，缺失则标记待确认。
- 奖励、概率、库存、道具 ID、奖励号缺失时必须列为待确认。
- QPS 不能脑补；PV/UV、峰值集中时间不足时，需要提示缺失。
- 页面多语言需要检查页面截图、字符限制、时间格式、时区展示、LQA 翻译和测试周期。
- 投放链接检查时，不主动改公式，只提示基础链接、渠道和参数是否缺失。
- 埋点检查需要遵守前后端埋点 Skill 的规则。
- 后端埋点只有用户明确说明“后端定制开发需求”时才应出现。
- 风险项需要说明可能影响配置、开发、测试、LQA、压测、域名发布、看板或复盘的哪个环节。
- 建议动作需要尽量明确负责人或确认方；缺失时标记“待补充”。

## forbidden_outputs

- 不要替需求方脑补最终奖励、概率、库存或任务规则。
- 不要脑补道具 ID、奖励号、活动时间、时区、QPS。
- 不要把待确认内容写成已确认。
- 不要只输出“文档不完整”这类笼统结论。
- 不要擅自新增同步文档中没有依据的业务规则。
- 不要直接判断活动一定可以进入下一阶段。
- 不要在未确认后端类型时判断需要后端埋点。
- 不要破坏或重写投放链接公式。
