# tracking-design_前后端埋点设计

## name

tracking-design_前后端埋点设计

## description

用于网页活动 / H5 活动的前后端埋点设计。前端埋点无论后端类型如何都需要生成；后端埋点只有在用户明确说明本次是“后端定制开发需求”时才生成。

## when_to_use

- 需要为网页活动 / H5 活动生成前端埋点时。
- 用户已明确本次后端类型，需要判断是否生成后端埋点时。
- 需要把活动埋点整理成可复制到产品文档或同步文档的表格时。
- 需要基于历史命名规则和历史格式补齐同类埋点时。

- 页面列表：待补充。
- 用户主流程：待补充。
- 页面模块和核心交互：待补充。
- 看板关注指标：待补充。
- 历史埋点命名规则：待补充。
- 历史埋点格式：待补充。

- 后端定制开发范围：仅在后端定制开发需求时需要提供，缺失则标记“待补充”。

- 说明：

如果用户未明确后端类型，则不要继续生成埋点表，直接输出确认问题：


## rules

### 后端需求类型处理规则

生成埋点前，必须先确认本次需求的后端类型。

| 后端需求类型 | 含义 | 是否需要写后端埋点 |
| --- | --- | --- |
| 后端配置需求 | 仅使用已有活动后台 / GM 后台 / 配置后台能力完成，例如配置奖励、概率、库存、任务、活动时间、开关、道具、礼包等 | 不需要写后端埋点 |
| 后端定制开发需求 | 本次活动需要后端新增接口、改造接口、新增业务逻辑、新增发奖逻辑、新增资格校验、定制任务计算、定制抽奖 / 兑换 / 支付 / 邀请关系等 | 需要写后端埋点 |

处理规则：

1. 前端埋点无论如何都需要写。
2. 后端埋点不是默认必写。
3. 只有当用户明确说明本次是“后端定制开发需求”时，才生成后端埋点。
4. 如果用户明确说明本次是“后端配置需求”，则只生成前端埋点，不生成后端埋点。
5. 如果用户没有明确说明后端类型，不要自行判断，不要猜测，不要根据“奖励、任务、抽奖、兑换”等玩法自动推断。
6. 如果用户未说明后端类型，需要先向用户提问确认：

```text
本次需求是后端配置需求，还是后端定制开发需求？如果只是后台配置奖励、概率、库存、任务、活动时间等，则不需要写后端埋点；如果涉及后端新增接口或定制业务逻辑，则需要写后端埋点。
```

7. 在用户确认后，再继续生成完整埋点设计。
8. 不要把后台配置项误判为后端定制开发项。
9. 不要因为活动涉及奖励、抽奖、兑换、任务，就自动生成后端埋点。

### 前端埋点生成规则

前端埋点无论后端类型如何，都需要生成。

前端埋点重点覆盖：

- 页面访问；
- 页面模块曝光；
- 按钮点击；
- 弹窗展示；
- tab 切换；
- 分享点击；
- 规则入口点击；
- 奖励说明入口点击；
- 任务入口点击；
- 抽奖入口点击；
- 兑换入口点击；
- 商城商品点击；
- 主要 CTA 点击。

### 后端埋点生成规则

后端埋点只有在用户明确说明“本次是后端定制开发需求”时才生成。

如果用户明确说明“本次是后端配置需求”，则不生成后端埋点。

如果用户没有说明后端类型，必须先向用户确认，不要直接输出后端埋点。

后端定制开发需求下，后端埋点重点覆盖：

- 新增接口的关键业务结果；
- 定制任务完成结果；
- 定制奖励发放结果；
- 定制抽奖结果；
- 定制兑换结果；
- 定制礼包发放结果；
- 定制支付 / 下单 / 购买结果；
- 定制邀请关系确认结果；
- 定制资格校验结果。

不要为以下后端配置项生成后端埋点：

- 后台配置奖励；
- 后台配置概率；
- 后台配置库存；
- 后台配置任务；
- 后台配置活动时间；
- 后台配置开关；
- 后台配置礼包；
- 后台配置道具；
- 后台配置活动规则。

### 通用埋点规则

- 如果用户提供命名规则和历史格式，必须严格沿用。
- 上报结构需要合并成一行，不要每行重复拆。
- 备注要简短，只写必要说明。
- 只输出与页面访问、核心点击、关键曝光、核心转化、定制后端业务结果相关的埋点。
- 对缺失信息标记“待补充”。

## forbidden_outputs

- 禁止在用户未明确后端类型时，自行判断后端类型。
- 禁止在用户未明确后端类型时，直接生成后端埋点。
- 禁止在只有后端配置需求时生成后端埋点。
- 禁止把后台配置项误写成后端埋点。
- 禁止因为活动涉及奖励、任务、抽奖、兑换，就默认生成后端埋点。
- 禁止把后端配置后台已有能力写成后端定制开发结果埋点。
- 禁止跳过后端类型确认，直接输出完整埋点表。
- 不要过度输出订单 ID、玩家 ID、设备 ID 等看板中不会使用的细参数。
- 不要输出无意义埋点。
- 不要把后端失败原因枚举写进前端或后端埋点。
- 不要为了显得完整而扩写大量待确认问题。
- 不要擅自改变用户提供的命名规则。
- 不要把未提供的信息写成已确认。
